## 💡 Proposal
Based on the project context, particularly the high latency in vector search (794.9ms) and the existence of wikilink analysis functions, I propose the following technical improvement:

### **Improvement: Graph-Weighted Context Reranking (GWCR)**

**The Problem:**
Currently, `proc_16_chat` relies on `get_context_for_query`, which utilizes a combination of BM25 and Vector search. While vector search is thorough, it is slow and lacks "structural awareness." A note might be semantically similar to a query but contextually irrelevant if it is an isolated "ghost link" or a low-value leaf node.

**The Solution:**
Integrate the existing logic from `get_wikilink_frequencies` (from `materialize_nodes.py`) directly into the reranking phase of `get_context_for_query`. 

Instead of relying solely on similarity scores, we should apply a **"Centrality Multiplier"** to the retrieved context.

**Technical Implementation:**
1.  **Extract Frequency Map:** In `graph_manager.py`, call `get_wikilink_frequencies` to get a map of how often each node is referenced across the vault.
2.  **Apply Scoring Heuristic:** When `get_context_for_query` returns the top $N$ candidates from the vector/BM25 search, adjust their final score using the formula:
    $$FinalScore = (SimilarityScore \times W_s) + (\log(LinkFrequency + 1) \times W_g)$$
    *(Where $W_s$ is similarity weight and $W_g$ is graph weight).*
3.  **Filter Ghost Links:** Use the logic from `remove_isolated_wikilinks` to penalize or exclude nodes that have zero incoming/outgoing links, as these are often incomplete thoughts that clutter the `build_system_prompt`.

**Why this is useful:**
*   **Relevance:** It prioritizes "Hub" notes—nodes that you have linked to many times—which usually contain foundational definitions or high-level summaries, making the AI's response more grounded in your core knowledge.
*   **Efficiency:** By reranking a smaller subset of vector results (e.g., the top 20), you improve the quality of the context sent to `build_system_prompt` without needing to run more expensive vector calculations.
*   **Synergy:** It leverages existing functions (`get_wikilink_frequencies`) that are currently isolated in `proc_22_main` and brings their value into the primary `proc_16_chat` user experience.
