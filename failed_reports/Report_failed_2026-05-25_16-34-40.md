## 💡 Proposal
To address the "Status 1" failure during a complex architectural query, I propose the following technical improvement:

### **Improvement: The "Architectural Skeleton" Fallback Mode**

**The Problem:**
When querying GitNexus about high-level patterns like "Hub & Spoke," the tool often fails (Exit Status 1) because it attempts to ingest the entire codebase to find an answer. This leads to context window overflows, API timeouts, or memory crashes when the project contains multiple "Spoke" repositories or large dependency trees.

**The Solution:**
Implement an **Adaptive Context Filter** that triggers an automatic "Skeleton Scan" upon a primary query failure. 

Instead of treating all files with equal weight, the tool should switch to a hierarchical extraction mode:
1.  **Level 1 (Structural):** Read only file names, folder structures, and `README.md` files.
2.  **Level 2 (Orchestration):** Identify and parse only "connector" files (e.g., `docker-compose.yml`, `terraform` files, `package.json` workspaces, or gateway configs).
3.  **Level 3 (Summarization):** Pass only this "skeleton" to the LLM to reconstruct the Hub & Spoke flow.

**Why this is a "Creative Leader" choice:**
It moves the tool from a binary "Success/Failure" state to a **"Graceful Degradation"** model. Even if the tool cannot process every line of code, it can still provide the architectural summary requested by focusing on the *topology* rather than the *telemetry* of the code.

**Technical Implementation:**
Add a retry logic in the `query` command:
```python
if status == 1 and "--skip-git" in args:
    trigger_fallback_mode(strategy="structural_only")
```
This ensures that the user receives their architectural summary based on the project's "blueprint" rather than getting stuck in the "engine room."
