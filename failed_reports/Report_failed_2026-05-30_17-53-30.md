Based on the current architecture and the performance bottlenecks identified in your timing data (specifically the 715ms vector search latency and the missing FTS indexes), I propose the following technical improvement:

### **Improvement: Graph-Weighted Hybrid Retrieval (GWHR)**

**The Concept:**  
Currently, `get_context_for_query` (in `proc_12_chat`) relies heavily on a slow vector search, while the faster keyword-based search (BM25) is degraded due to missing indexes. Furthermore, the system already possesses a tool to measure graph connectivity via `get_wikilink_frequencies`. 

This improvement integrates these three elements into a single, high-performance retrieval pipeline that uses the **knowledge graph's topology** to refine semantic search results.

---

### **Technical Implementation**

1.  **Automated FTS Maintenance Middleware:**  
    Modify the initialization of `graph_manager.py` to check for FTS index integrity on startup. If missing (addressing the "FTS indexes missing" warning), it should trigger a background re-index of the wikilink structures to restore the 16.2ms BM25 performance.

2.  **The Reranking Algorithm:**  
    Update `get_context_for_query` to implement a **Reciprocal Rank Fusion (RRF)** score that combines Vector, BM25, and a new **"Graph Density Boost."**
    *   **Vector Score ($V$):** Semantic similarity (current bottleneck).
    *   **Keyword Score ($K$):** Fast BM25 score.
    *   **Connectivity Weight ($W$):** Derived from `get_wikilink_frequencies`. 

    **The Formula:**  
    $Score(node) = \frac{1}{k + Rank_V} + \frac{1}{k + Rank_K} + (\text{normalized } \textit{wikilink\_frequency} \times \alpha)$

3.  **Contextual Pruning via `validator.py`:**  
    Inject the logic from `_rule_has_wikilinks` and `remove_isolated_wikilinks` into the retrieval step. Instead of just finding similar text, the system will prioritize "Pillar Nodes"—nodes that are semantically relevant *and* serve as central hubs in the graph.

---

### **Why this is useful:**

*   **Mitigates Latency:** By utilizing the fast BM25 (16ms) alongside vector search, the system can provide a "fast-path" response or a better-ranked "slow-path" response, making the 715ms vector wait more worthwhile.
*   **Contextual Integrity:** It prevents "Ghost Links" (identified in `remove_ghost_links.py`) from being surfaced as relevant context, ensuring the AI only receives well-connected, validated information.
*   **Improved RAG Accuracy:** In a knowledge graph (Autobrain), semantic similarity isn't enough. A node with 50 incoming wikilinks is objectively more "important" to the system's core logic than an isolated leaf node. This improvement ensures the AI "thinks" from the perspective of the most established concepts in your vault.
