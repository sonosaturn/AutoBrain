## 💡 Proposal
Based on the project context and the performance metrics provided, I propose the following technical improvement:

### **Improvement: Graph-Weighted Context Reranking**

**The Problem:**
Currently, `proc_12_chat` relies on `get_context_for_query` which utilizes a vector search taking **796.5ms** (approx. 86% of the total wall time). While vector search is excellent for semantic similarity, it ignores the structural importance of information within your knowledge graph. Furthermore, your keyword search (FTS) is currently degraded, placing the entire burden of relevance on semantic embeddings.

**The Proposal:**
Integrate the logic from `proc_24_main` (`get_wikilink_frequencies`) directly into the `get_context_for_query` pipeline in `autobrain_core/graph_manager.py`. 

Instead of returning context based solely on vector similarity, implement a **re-ranking algorithm** that weights results using "node centrality" (based on wikilink frequencies).

**Technical Implementation:**
1.  **Extract Centrality:** Use the existing `get_wikilink_frequencies` function to generate a map of "authority" nodes (nodes frequently linked to by others).
2.  **Modify `get_context_for_query`:** 
    *   Retrieve the top $N$ candidates via vector search.
    *   Apply a boosting factor to the similarity score of each candidate based on its wikilink frequency. 
    *   $Score_{final} = (w_1 \cdot Similarity_{vector}) + (w_2 \cdot \log(Frequency_{link} + 1))$
3.  **Optimization:** Cache the wikilink frequency map in memory since it only needs to update when `materialize_nodes.py` runs, avoiding additional overhead during the chat loop.

**Why this is "Creative Leader" material:**
*   **Contextual Intelligence:** It moves the system from "simple search" to "structural awareness." The AI will prioritize "Hub" notes (high-frequency links) which usually contain foundational definitions, making the `build_system_prompt` significantly more coherent.
*   **Performance Mitigation:** Since vector search is slow, re-ranking a smaller set of vector results with graph data allows you to reduce the number of high-dimensional vectors you need to compare without losing retrieval quality.
*   **Synergy:** It bridges the gap between the `autobrain_core` graph maintenance tools and the `jarvis` user-facing chat interface.

**Immediate Next Step:**
Fix the missing FTS indexes as noted in the warning (`gitnexus analyze --force`). Once FTS is restored, the re-ranking should be applied to a **Hybrid Search** (Vector + FTS) to further stabilize the retrieval against the 796ms latency.
